### Web UI: access paths and integration options

- Previously, clicking AskTIM opened a UI inline within the course site. With an external site, students navigate away.
- Options considered for how students reach the tutor:
  1. Add a direct link alongside each exercise.
  2. Add a link on the course introduction page; tutor asks which exercise the student is on when they arrive.
  3. Do both — students tend to follow the path of least friction.
  4. **Split view**: external website shows the exercise on one side and the tutor on the other.
  5. **iFrames**: if the engineering team allows iFrames on the course website, the tutor can be embedded internally without redirecting students.
- iFrames appear to be feasible. Faizan will confirm by EOD whether the iFrames Tool can be enabled for this course; decision on which path to pursue will be made tomorrow.
- Preferred path: iFrames integration.
- Fallback: split view (exercise + tutor side-by-side). Notes on split view:
  - Some exercises include diagrams not currently in the tutor's context; need to confirm whether diagrams are important before Week 4 when those exercises begin.
  - If iFrames are not available on the external site, MITLearn could be embedded as an iframe there instead — requires testing across screen sizes.
  - Simpler fallback: show only the exercise text (not the full learn website) in the split view.
- Postgres implementation is the priority for the external site since conversations must be logged.

### Student identity and conversation logging

- Not currently possible to identify which student is accessing the tutor; Faizan will ask the engineering team — likely the same technical barrier as chat logging.
- Logging conversations tied to individual students is important: it enables tracking how students improve and engage across the semester, and is a compelling data point for MIT.
- Proposed workaround: after three messages, display a popup asking for the email address the student used to sign up for the class. Require a valid format (@ and .) before proceeding. Store the email in a cookie to avoid repeated prompts. A wrong email is not critical — connections are still useful even if imperfect.

### Conversation history and UX

- When a student clicks the tutor link, a new conversation is created with the exercise as context; previous conversations are shown as history.
- Using our own framework enables features not available through AskTIM: cross-exercise history, previous-session context, longitudinal tracking. Demonstrating these to MIT could push them to adopt similar features in AskTIM.
- Priority is to deploy something by the end of Week 2 so students adopt the tutor early in the 13-week semester; updates can follow incrementally.
- Avoid forcing students to the external site (e.g. by withholding exercise content); encourage adoption through TAs, professors, and frictionless access paths instead.
