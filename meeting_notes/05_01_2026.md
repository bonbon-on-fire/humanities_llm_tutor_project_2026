### Context and hallucination problem

- Without course context, the tutor hallucinates what the student has been taught so far.
- Need to add full context: class overview (high-level course goal), syllabus (lectures and their topics), and eventually readings and lecture notes.
- Hope is that providing much more context will raise the ceiling of what the tutor can do.
- The models are smart enough to handle a large amount of context without requiring too much hand-holding through it.
- Need to figure out a framework for adding context that is resilient to scaling across courses.
- For the AskTim integration: when a student clicks AskTim, it is already attached to a specific exercise, so the tutor will not need to ask which assignment. For standalone deployment the tutor can open by asking which assignment.

### Tool calls and lecture note retrieval

- Implement tool calls so the tutor can decide between answering the student directly or fetching relevant context from lecture notes.
- This may fall under the engineering team's responsibility — unless they provide an API to query their content, they would need to do it.
- MITLearn provides video transcripts and has an existing API that we can query. OCW does not have this, so the approach is platform-dependent.
- Even if the engineering team is not willing to make these changes, we can build our own simulation as a proof of concept for publication, which may convince them.

### May 12 shippable scope

- Do not add lecture notes as context for the May 12 launch — not enough time to build tool calls and get the engineering team to integrate.
- Add the syllabus as context by May 12, treating it the same way as course and exercise context currently provided.
